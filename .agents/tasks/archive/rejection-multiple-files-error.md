## Goal

A `*rejections.proto` with `java_multiple_files = true` (and/or a non-`Rejections`
`java_outer_classname`) must fail the build with a **clear, actionable** error message,
not a bare `java.lang.IllegalArgumentException (no error message)` (issue #38).

## Context

Rejection codegen requires the rejection *messages* to be nested inside a single outer
class (`...Rejections`) so that the top-level throwable classes generated by
`RThrowableRenderer` do not clash with the message classes of the same simple name.

`RThrowableRenderer.findRejectionFiles()` already called a private `checkConventions()`
that threw via `check()` (an `IllegalStateException`) with a serviceable message, but:

- It was **untested** — no regression coverage existed for the violation paths,
  which is why the issue stayed open.
- It read the option manually (`optionList.find("java_multiple_files", BoolValue...)`)
  instead of the idiomatic `header.javaMultipleFiles()` helper used elsewhere
  (`OuterClassAnnotationDiscovery`).
- The message did not state the *current* value or the precise fix.

## Plan

- [x] Extract the convention checks into `RejectionConventions.kt` as a
      `@VisibleForTesting public fun ProtobufSourceFile.checkRejectionConventions()`
      so the `signal-tests` module (separate Gradle module, public API only) can
      unit-test them.
- [x] Use idiomatic `header.javaMultipleFiles()` / `header.javaOuterClassName()`.
- [x] Throw `IllegalArgumentException` (invalid user *input*) with a message that names
      the file, the offending option, its value, and the remedy.
- [x] `RThrowableRenderer` calls the extracted function; drop the moved code and the
      now-unused `BoolValue` / `find` / `javaOuterClassName` imports and `RejectionFile`
      typealias.
- [x] Add `RejectionConventionsSpec` (signal-tests) covering: multiple-files true
      (throws), false/absent (ok); outer class non-`Rejections` (throws),
      `Rejections`/absent (ok); assert the message content.
- [x] Run `:signal-tests:test` and confirm green.
- [x] Verify the message surfaces in a real pipeline (RThrowableRenderer is reached
      before any other failure when multiple-files is on).

## Log

- 2026-06-15 — investigated; confirmed the detection logic is equivalent to the
  idiomatic helper, so the gap is message quality + missing tests. Implemented the
  refactor and the unit spec; build running to validate.
- 2026-06-16 — `:signal-tests:test` green: 6/6 in `RejectionConventionsSpec`,
  21/21 across the module (no regressions from the `signal` refactor).
  End-to-end reachability confirmed by inspection: with `java_multiple_files = true`
  the SignalPlugin renderers resolve the (separate-file) rejection message paths via
  the library's `MessageOrEnumConvention`, which explicitly branches on
  `javaMultipleFiles()` and does not throw — so `RThrowableRenderer` runs and its
  convention check is the first failure, surfacing the new message.
